

# AI开发规范（全局版）

## 一、开发原则

### 1. 开发前必须分析项目

任何代码编写前，必须先阅读项目代码并完成以下工作：

* 理解项目整体架构
* 理解项目目录结构
* 理解业务逻辑
* 理解数据库结构
* 理解前后端交互流程
* 理解已有编码规范
* 阅读README.md
* 所有配置内容写在app_config.json中，如果没有就先创建

禁止：

* 未阅读代码直接开发
* 未了解业务直接修改代码
* 未分析项目结构直接新增功能

---

### 2. 二次开发原则

如果是在已有项目上进行开发：

必须沿用原项目技术栈。

例如：

* 原项目使用 Vue，禁止改成 React
* 原项目使用 React，禁止改成 Vue
* 原项目使用 Java，禁止改成 Python
* 原项目使用 MySQL，禁止改成其他数据库

除非用户明确要求更换技术栈。

---

### 3. 修改最小化原则

开发时优先采用最小修改方案。

要求：

* 尽量复用现有代码
* 尽量复用现有组件
* 尽量复用现有接口
* 尽量复用现有数据库结构

禁止：

* 为了实现简单功能重构整个项目
* 大面积删除已有代码
* 无必要修改核心架构

---

### 4. 操作前说明原则

涉及以下操作时，必须先输出执行计划并等待用户确认：

* 删除数据库表
* 删除数据库数据
* 修改数据库结构
* 修改服务器配置
* 修改 Nginx 配置
* 修改 Systemd 配置
* 重启服务
* 重启服务器
* 批量修改数据

确认后才能执行。

---

## 二、README规范

### 1. 项目分析后必须更新 README.md

README 必须包含：

#### 项目介绍

说明项目用途与业务目标。

#### 项目目录结构

示例：

```text
src/
├── api/
├── components/
├── pages/
├── utils/
└── router/
```

并添加中文说明。

#### 技术栈说明

前端：

* HTML
* CSS
* JavaScript
* Vue
* React

后端：

* Node.js
* Python
* Java

数据库：

* MySQL

服务器：

* Ubuntu

#### API接口说明

列出：

* 接口地址
* 请求方式
* 参数说明
* 返回结果

#### 数据库说明

列出：

* 数据库名称
* 数据表名称
* 表用途说明

#### 功能模块说明

列出：

* 模块名称
* 模块用途
* 使用方法

#### 更新记录（ChangeLog）

每次开发后必须追加记录。

格式：

```text
2026-06-08

新增：
- 维修管理模块

修改：
- 财务统计页面

数据库：
- cars表新增repair_cost字段
```

---

### 2. 功能开发完成后

每新增一个功能：

必须更新 README。

说明：

* 功能作用
* 使用方法
* 涉及页面
* 涉及接口
* 涉及数据库
* 注意事项

## 五、启动脚本规范

所有后端项目必须提供：

```bash
start.sh
```

### 启动逻辑

1. 检查服务是否运行
2. 优先通过 PID 停止进程
3. PID不存在时通过端口查找
4. 清理旧进程
5. 启动服务
6. 检查启动状态
7. 输出启动结果

### 财务计算字段规范

1. 新增数据时
   前端/后端根据公式先计算出公式字段，并在页面显示给人员核对。

2. 人员点击保存时
   将原始录入字段保存到数据库。
   如果人员确认了公式结果，也将确认后的公式字段值保存到数据库。

3. 页面展示时
   如果数据库中的人工确认值/人工修改值有值，优先显示这个值。
   如果数据库中没有确认值，就通过视图实时计算并显示。

4. 人员修改公式值时
   点击保存后，将修改后的值保存到数据库。
   同时建议保存修改原因、修改人、修改时间。
