

# AI开发规范（全局版）

## 一、开发原则

### 1. 开发前必须分析项目

任何代码编写前，必须先阅读项目代码并完成以下工作：

* 理解项目整体架构
* 理解项目目录结构
* 理解业务逻辑
* 理解数据库结构
* 理解前后端交互流程
* 理解已有编码规范
* 阅读README.md
* 所有配置内容写在app_config.json中
* 我们是租车行业，在一些专有名词中注意多音字，如同行，应读tong hang
* 测试app_config.json给的数据库配置是否能连接

禁止：

* 未阅读代码直接开发
* 未了解业务直接修改代码
* 未分析项目结构直接新增功能

---

### 2. 二次开发原则

如果是在已有项目上进行开发：

必须沿用原项目技术栈。

例如：

* 原项目使用 Vue，禁止改成 React
* 原项目使用 React，禁止改成 Vue
* 原项目使用 Java，禁止改成 Python
* 原项目使用 MySQL，禁止改成其他数据库

除非用户明确要求更换技术栈。

---

### 3. 修改最小化原则

开发时优先采用最小修改方案。

要求：

* 尽量复用现有代码
* 尽量复用现有组件
* 尽量复用现有接口
* 尽量复用现有数据库结构

禁止：

* 为了实现简单功能重构整个项目
* 大面积删除已有代码
* 无必要修改核心架构

---

### 4. 操作前说明原则

涉及以下操作时，必须先输出执行计划并等待用户确认：

* 删除数据库表
* 删除数据库数据
* 修改数据库结构
* 修改服务器配置
* 修改 Nginx 配置
* 修改 Systemd 配置
* 重启服务
* 重启服务器
* 批量修改数据

确认后才能执行。

### 计算字段规范

涉及公式计算的字段，按“基础表保存原始录入数据和人工确认值，视图/后端负责实时计算，前端负责展示核对和保存”的规则处理。

核心原则：

- 公式字段不是只读展示字段。
- 公式字段必须允许用户核对、确认、手动修改并保存。
- 前端不能因为字段配置了公式，就把该字段固定渲染为纯文本。
- 只有数据库字段本身不可编辑时，公式字段才允许只读展示。

1. 新增数据时：
   前端或后端先根据公式计算出结果，并在前端页面展示给用户核对。页面上的公式字段应显示为可编辑输入控件，让用户可以直接确认或修改。

2. 数据库展示时：
   可以通过数据库视图实时计算公式字段，并展示完整表数据。视图只负责提供默认计算值，不代替基础表中的人工确认/修改值。

3. 用户点击保存时：
   - 原始录入字段保存到基础表。
   - 如果用户确认了公式结果，将确认后的公式字段值保存到基础表对应的确认字段。
   - 如果用户手动修改了公式结果，将修改后的值保存到基础表对应的确认字段。
   - 公式字段只要前端页面当前展示了值，保存时就应按用户确认值处理，写入基础表对应字段。

4. 页面再次展示时：
   - 如果基础表中已有用户确认值或人工修改值，优先显示该值。
   - 如果基础表中没有确认值，则使用数据库视图或后端公式实时计算结果。
   - 使用公式实时计算结果展示时，前端仍应保持该字段可编辑，方便用户覆盖默认计算值。

5. 人员修改公式值时：
   点击保存后，将修改后的值保存到数据库。

6. 字段职责：
   - 基础表：保存原始录入字段，以及人工确认/修改后的公式值。
   - 视图：负责实时计算默认公式值，并展示完整表数据。
   - 前端：负责展示公式结果、让用户核对、允许用户修改、提交保存。

7. 保存兜底：
   - 前端保存时应提交公式字段当前展示值。
   - 后端保存时也必须根据公式重新计算缺失的公式字段值，并写入基础表，不能只依赖前端提交。
   - 后端：负责校验公式配置、生成或同步公式视图，并在保存时把确认后的公式字段值写入基础表。

7. 展示优先级：

人工确认/修改值 > 视图实时计算值 > 空值

8. 禁止事项：
   - 禁止把可编辑公式字段做成只读文本。
   - 禁止只保存原始字段而丢弃公式字段当前展示值。
   - 禁止保存时批量覆盖已有人工确认/修改值，除非用户明确修改了这些值。
   - 禁止让视图计算值覆盖基础表中已有的人工确认/修改值。


### 单选和多选字段规范

字段类型以企业微信智能表字段配置为准。

1. 单选字段：
   - 数据库字段使用 MySQL `ENUM`。
   - `ENUM` 的选项必须和企业微信智能表单选项保持一致。
   - 前端使用单选下拉框、单选按钮或同类单选组件。
   - 前端显示中文选项，数据库保存对应的中文选项值。

2. 多选字段：
   - 数据库字段使用 MySQL `JSON`。
   - 数据库保存 JSON 数组，不保存逗号拼接字符串。
   - JSON 示例：

```json
["门店", "维修厂"]
```

3. 前端选择多选字段时：
   - 使用多选下拉框、复选框组或标签选择器。
   - 用户可以同时选择多个选项。
   - 前端表单值使用数组。

```js
form.current_location = ["门店", "维修厂"]
```

4. 保存多选字段时：
   - 前端提交数组。
   - 后端校验每个值是否存在于企业微信智能表字段配置的选项中。
   - 校验通过后写入数据库 JSON 字段。

5. 展示多选字段时：
   - 后端返回 JSON 数组。
   - 前端解析为数组后显示多个标签或多个已选项。
   - 页面上显示中文选项，不直接显示 JSON 字符串。

6. 查询多选字段时：
   可以使用 MySQL JSON 查询函数，例如：

```sql
JSON_CONTAINS(current_location, JSON_QUOTE('门店'))
```

7. 已执行的数据库规则：
   - 企业微信未隐藏表中的单选字段已改为 `ENUM`。
   - 企业微信未隐藏表中的多选字段已改为 `JSON`。


### 字段关联选项规范

当一个表的字段需要从另一个表中选择值时，必须使用通用“字段选项来源”配置，不允许为某个字段写死特殊逻辑。

1. 配置入口：
   - 在表格管理中选择目标表。
   - 找到目标字段，点击字段配置。
   - 选择“从数据表读取”。
   - 选择来源表和来源字段。

2. 配置保存位置：
   - 表名和字段名保存在 `cw_field_option_config.table_name`、`cw_field_option_config.column_name`。
   - 固定选项保存在 `cw_field_option_config.options_json`。
   - 数据表来源保存在 `cw_field_option_config.option_source_type`、`source_table_name`、`source_column_name`。

3. 通用要求：
   - 任意业务表字段都可以配置从任意业务表字段读取选项。
   - 系统字段如 `id`、`create_time`、`update_time`、`status` 不作为普通业务选项字段配置。
   - 图片、文件、附件类字段不应配置为下拉选项。
   - 来源字段读取时应去重、过滤空值。

4. 前端展示：
   - 配置了来源表字段的字段，在中台中显示为下拉选择。
   - 页面保存时，保存用户选择后的字段值到当前业务表。

5. 示例：
   - 报销费用明细表 `cw_bxfymxb.bxr` 从人员表 `cw_ryb.display_name` 读取报销人选项。
   - 后续其他字段如负责人、销售、车队长、客户名称、车牌号等，也使用同一套配置方式。


### 字段类型标签规范

表格管理中字段类型为“文本”或“日期”的字段不显示标签，其他字段类型显示标签。

当前可显示和更改的字段类型标签包括：

- 单选
- 多选
- 图片
- 文件
- 关联
- 计算

规则：

1. 标签用于控制字段的前端展示和后续字段行为。
2. 标签配置保存在 `cw_field_option_config.field_kind`。
3. 字段未配置标签时，系统可根据数据库字段类型、字段名、公式配置等信息自动推断。
4. 点击标签时弹出类型菜单，由管理员选择字段类型。
5. 只有配置为“计算”的字段显示“公式”按钮。
6. 只有配置为“单选”“多选”“关联”的字段显示“配置”按钮。
7. 配置为“计算”的字段，应配合公式配置使用。
8. 配置为“图片”或“文件”的字段，应走图片/文件上传和 JSON 存储规范。
9. 配置为“关联”的字段，应优先通过“从数据表读取”配置来源表和来源字段。

### 图片/文件字段规范

字段类型配置为“图片”或“文件”后，中台表格中点击该字段应打开强模态弹窗。

1. 弹窗打开后，页面其他区域不可操作。
2. 弹窗右上角提供关闭按钮。
3. 弹窗底部提供“上传”“下载”“删除”“保存”按钮。
4. 图片字段只能上传图片 MIME 类型。
5. 文件字段不能上传图片 MIME 类型，图片必须使用图片字段上传。
6. 图片字段上传时，前端必须先将图片转换为 WebP 格式后再保存，最长边压缩到 1600px 以内，文件名使用 `.webp` 后缀，下载时也下载 WebP 文件。
7. 文件字段保持原文件格式保存和下载。
8. 上传后的文件以数组形式保存，元素包含文件名、类型、大小、data URL 和上传时间。
9. 点击“保存”时，将弹窗中的文件数组写回当前单元格，并进入中台表格原有保存流程。
10. 中台表格保存接口允许较大的 JSON 请求体，用于承载图片/文件字段数据。
11. 图片字段弹窗中点击图片，应使用历史图片预览同款样式打开预览，预览标题显示图片文件名，底部不显示推送按钮。
12. 文件字段弹窗中点击文件，应打开文件预览弹窗，预览标题显示文件名，底部提供下载按钮；Excel/CSV 渲染为表格预览，PDF、文本等浏览器可内嵌类型直接预览，Word、压缩包等不强制预览，显示下载提示。


### 图片及文件存放方式

1. 图片和文件都存放在服务器本地目录 `create_file/` 下。

2. 数据库不存图片/文件本体，只存文件信息 JSON。

### 审计日志规范

系统必须维护两张不带 `cw_` 前缀的审计日志表。

1. `user_login_log`：
   - 记录用户登录行为。
   - 字段应包含登录时间、用户 ID、账号、显示名、权限。

2. `middle_operation_log`：
   - 记录中台表格操作行为。
   - 字段应包含操作时间、用户 ID、账号、显示名、操作表名、表显示名、操作类型、主键字段、主键值、变更内容。
   - 操作类型为 `insert`、`update`、`delete`，分别对应新增、修改、删除。
   - 操作表名记录真实业务表名，例如 `cw_bxfymxb`；日志表自身不使用 `cw_` 前缀。
   - 删除操作的变更内容必须保存删除前的整行快照，格式为 `{ "before": {...} }`。

### 中台表格排序规范

中台表格默认按旧到新显示。

1. 有 `create_time` 和 `id` 的表，按 `create_time ASC, id ASC` 排序。
2. 只有 `create_time` 的表，按 `create_time ASC` 排序。
3. 只有 `id` 的表，按 `id ASC` 排序。
4. 新增数据保存后应出现在最后一行。

### 中台分页规范

1. 中台表格每页固定显示 500 条数据。
2. 新增、删除按钮后方显示分页控件。
3. 分页控件包含上一页箭头、页码输入框、总页数、下一页箭头。
4. 点击左右箭头向前或向后翻页。
5. 点击页码输入框可直接输入页码，按回车跳转到对应页。

### 中台 ID 显示规范

1. 中台表格最左侧固定列显示真实表主键 ID，不显示页面序号。
2. `id` 字段可在表格管理中配置。
3. 即使 `id` 被配置为显示字段，中台普通字段区域也不重复显示主键列，主键只在最左侧固定列显示。

3. 图片和文件字段都支持多个文件，数据库字段存 JSON 数组。

4. JSON 格式如下：

[
  {
    "name": "原文件名.pdf",
    "url": "/create_file/cw_swpzb_pzfj_file/20260609/唯一文件名.pdf",
    "type": "application/pdf",
    "size": 123456,
    "uploadedAt": "2026-06-09 18:00:00"
  }
]

5. 存放目录命名规则：

表名_字段名_文件类型

文件类型：
- 图片字段使用 img
- 普通文件字段使用 file

示例：
- 维修报备申请表的修理厂A报价：
  create_file/cw_wxbbsqb_xlcabj_img/
- 税务凭证的凭证附件：
  create_file/cw_swpzb_pzfj_file/

6. 后端上传文件时：
- 根据表名、字段名、文件类型找到对应目录
- 如果目录不存在则创建
- 文件名必须生成唯一名称，不能直接使用原文件名
- 保存成功后返回 JSON 文件信息

7. 前端展示时：
- 图片字段显示缩略图，点击可预览
- 文件字段显示文件名，点击可下载
- 支持重新上传、下载、删除

8. 删除文件时：
- 删除服务器上的文件
- 同步更新数据库字段中的 JSON
